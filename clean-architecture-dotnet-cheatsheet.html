Clean Architecture in .NET Web API

An enhanced cheatsheet covering core principles, layer responsibilities, project structure, C# examples, and best practices for building robust and maintainable .NET Web APIs using Clean Architecture.

Last Updated: June 7, 2025

Core Concepts

Core Philosophy
  • The Dependency Rule: The cornerstone. Promotes independence of core business logic.
  • Abstraction: Inner layers define interfaces, outer layers implement them.
  • Testability: Business logic can be tested in isolation.
  • Maintainability & Flexibility: Easier to evolve the system with minimal impact across layers.
When to Use Clean Architecture
  • Complex Business Logic: When core rules are intricate and form the primary value.
  • Long-Term Projects: For applications expected to evolve and be maintained.
  • Domain-Driven Design (DDD) Alignment: When focusing on the domain model.
  • High Testability Requirements: When isolated testing of business logic is crucial.
  • Technology Independence: To defer or easily change decisions about frameworks, databases, or UI.

Layers Overview & Responsibilities

Domain Layer (Core)

Purpose: Contains enterprise-wide business logic and types. Innermost layer with no dependencies on other solution layers.

Key Components:
  • Entities: Core business objects with identity and logic. 
    // Domain/Entities/Product.cs
public class Product
{
    public Guid Id { get; private set; }
    public string Name { get; private set; }
    public decimal Price { get; private set; }

    // Constructor and methods with business logic...
    public Product(Guid id, string name, decimal price) { /* ... validation ... */ }
    public void UpdatePrice(decimal newPrice) { /* ... logic ... */ }
}
  • Value Objects: Immutable descriptive aspects (e.g., Address).
  • Aggregates: Clusters of domain objects treated as a unit.
  • Domain Services: Logic not fitting a single entity.
  • Repository Interfaces: Data access abstractions. 
    // Domain/Interfaces/IProductRepository.cs
public interface IProductRepository
{
    Task<Product?> GetByIdAsync(Guid id, CancellationToken cancellationToken = default);
    Task<IEnumerable<Product>> GetAllAsync(CancellationToken cancellationToken = default);
    Task AddAsync(Product product, CancellationToken cancellationToken = default);
    Task UpdateAsync(Product product, CancellationToken cancellationToken = default);
    Task DeleteAsync(Guid id, CancellationToken cancellationToken = default);
}
  • Domain Events: Represent significant occurrences.
  • Custom Domain Exceptions: Signal domain rule violations.
Application Layer

Purpose: Contains application-specific business logic. Orchestrates use cases, interacting with Domain and Infrastructure (via interfaces).

Key Components:
  • Use Cases/Interactors/Services: Application operations. 
    // Application/Products/Commands/CreateProductCommand.cs
public record CreateProductCommand(string Name, decimal Price) : IRequest<Guid>;

// Application/Products/Handlers/CreateProductCommandHandler.cs
public class CreateProductCommandHandler : IRequestHandler<CreateProductCommand, Guid>
{
    private readonly IProductRepository _productRepository;
    private readonly IUnitOfWork _unitOfWork;

    public CreateProductCommandHandler(IProductRepository productRepository, IUnitOfWork unitOfWork)
    {
        _productRepository = productRepository;
        _unitOfWork = unitOfWork;
    }

    public async Task<Guid> Handle(CreateProductCommand request, CancellationToken cancellationToken)
    {
        var product = new Product(Guid.NewGuid(), request.Name, request.Price);
        await _productRepository.AddAsync(product, cancellationToken);
        await _unitOfWork.SaveChangesAsync(cancellationToken);
        return product.Id;
    }
}
  • Data Transfer Objects (DTOs): Data carriers between layers.
  • Interfaces for Infrastructure Services: Abstractions for external concerns (e.g., `IEmailSender`, `IDateTimeProvider`).
  • Validation Logic: Input data validation (e.g., using FluentValidation).
  • CQRS: Separates read and write operations. Often uses MediatR.
  • Application Exceptions: For application-specific errors (e.g., `NotFoundException`).
Infrastructure Layer

Purpose: Handles all external concerns and technical details (databases, file systems, external APIs). Implements interfaces from Application/Domain.

Key Components:
  • Data Access/Persistence: Repository implementations (e.g., using Entity Framework Core). Includes `DbContext`.

public class ProductRepository : IProductRepository

    public ApplicationDbContext(DbContextOptions<ApplicationDbContext> options) : base(options) { }

    // ... other DbSets & OnModelCreating

  • External Service Integrations: Clients for APIs, payment gateways, email.
  • Caching Implementations: (e.g., Redis, In-Memory).
  • Identity Services: Authentication/Authorization implementations.
  • File System Access, Clock Services, etc.
Presentation Layer (Web API)

Purpose: Handles user interaction (HTTP requests/responses for an API). Translates input to Application layer and presents results.

Key Components:
  • Controllers/API Endpoints: Receive HTTP requests, delegate to Application layer. 
    // Presentation/Controllers/ProductsController.cs
[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    private readonly ISender _mediator; // MediatR
    public ProductsController(ISender mediator) {/*...*/}

    [HttpPost]
    public async Task<IActionResult> CreateProduct([FromBody] CreateProductCommand command)
    {
        var productId = await _mediator.Send(command);
        return CreatedAtAction(nameof(GetProductById), new { id = productId }, new { id = productId });
    }

    [HttpGet("{id}")]
    public async Task<IActionResult> GetProductById(Guid id) { /* ... */ }
}
  • Middleware: Global error handling, authentication, logging.
  • Dependency Injection (DI) Setup: Configuration of services (Composition Root).
  • API Models/ViewModels: Models for API requests/responses.
  • API Versioning, OpenAPI/Swagger Configuration.

Best Practices & Considerations

Key Practices
  • Dependency Injection (DI): Absolutely crucial. Register dependencies in Presentation layer's `Program.cs`. Use constructor injection.
  • MediatR: Widely used for implementing CQRS in Application layer. Decouples command/query senders from handlers. Allows cross-cutting concerns via pipeline behaviors (validation, logging).
  • FluentValidation: Robust validation in Application layer, often integrated with MediatR pipelines.
  • AutoMapper (or similar): Useful for mapping between Entities, DTOs, and API Models. Define profiles where relevant.
  • Unit of Work (UoW) Pattern: Often implemented in Infrastructure (e.g., `DbContext`). An `IUnitOfWork` interface can be defined in Application layer.
  • Error Handling: Implement global error handling middleware in Presentation layer. Define custom exceptions in Domain and Application layers.
  • Configuration (Options Pattern): Manage configuration via `IOptions`, configured in Presentation layer.
  • Async/Await: Use thoroughly for I/O-bound operations (database access, network calls) to maintain responsiveness.
  • Single Responsibility Principle (SRP): Apply SRP to classes and methods within each layer.
  • Lean Controllers: Controllers should be thin, primarily delegating work to the Application layer. Avoid business logic in controllers.
  • Avoid Leaking Abstractions: Do not expose `IQueryable` from repositories to outer layers, as this can lead to infrastructure concerns leaking. Queries should be fully defined within the data access layer or use specification patterns.
  • Testing Strategy: Unit tests for Domain & Application layers (mocking dependencies). Integration tests for Infrastructure & Presentation layers (testing against real DBs/services or in-memory test servers).
