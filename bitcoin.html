// II. Core Protocols

Interface Definition

Software/Hardware interface managing cryptographic keys. Enables TX signing & balance lookup. Keys, not coins, are stored.

Key Pairs :: Asymmetric Crypto

Private Key: Secret. Signs TXs. [GUARD]. Public Key: Derives Addr. Shareable.

Addresses :: Receiving Nodes

Receiving Identifiers. Formats: Legacy(1), Nested SegWit(3), Native SegWit(bc1q), Taproot(bc1p).

Address Type Analysis
  • Legacy (P2PKH): Max compatibility, highest fees.
  • Nested SegWit (P2SH): Wrapped SegWit, better fees. Bridge format.
  • Native SegWit (Bech32): Lowest fees, best error check. Current standard.
  • Taproot (Bech32m): Latest upgrade. Privacy/scripting benefits. Future standard.

OPSEC: Generate new address per TX for privacy enhancement.

Seed Phrase :: Genesis Code (BIP-39)

12/24 word mnemonic -> Master Key. [ULTIMATE BACKUP PROTOCOL]. (BIP-39 Spec).

Critical Function & Derivation
  • Function: Sole recovery vector post-device failure/loss.
  • Origin: Secure generation via wallet RNG (offline preferred).
  • Derivation Path: Uses BIP-32 + BIP-44/84/86 standards to generate key hierarchy.
  • Seed -> Master Key -> Tree of Keys -> Addresses
  • Tool (Offline ONLY): Ian Coleman BIP39 [WARNING: Extreme caution required if used]
TX Flow :: Network Propagation

Consume UTXOs -> Create Outputs. Attach fee (sat/vB). Broadcast to Mempool -> Await Mining Confirmation.

Transaction Lifecycle:
  • UTXOs: Unspent outputs; represent ownable BTC units.
  • Inputs/Outputs: TX consumes UTXOs, creates new ones (recipient + change).
  • Fees (sat/vB): Market-based incentive for miners; size-dependent.
  • Mempool: P2P queue for unconfirmed TXs.
  • Confirmations: Block inclusion = finality increases. (6+ = standard security).
  • Monitor Node: mempool.space

// III.A Hot Interfaces :: ONLINE/NETWORKED

Software Terminals

Applications (Desktop/Mobile). Keys reside on networked device = High Attack Surface. Suitable for operational funds only.

Custody Protocol Distinction:
  • Non-Custodial: User controls keys/seed. [Self-Sovereign]
  • Custodial: Third-party controls keys. [Counterparty Risk]
Non-Custodial Access Nodes:
  • Desktop: Electrum, Sparrow, Specter
  • Mobile: BlueWallet, Muun, Samourai(*Note Legal Status*)
Threat Assessment:
  • Vulnerable to: Malware, Phishing, OS Exploits, Physical Device Compromise.
Web Interfaces

Browser-based access nodes. Often custodial. EXTREME RISK PROFILE via XSS, Phishing, Browser Malware.

// III.B Cold Storage :: OFFLINE/ISOLATED

Hardware Modules

Dedicated crypto-processors. Key isolation from network. Maximum Online Threat Resistance.

Core Functionality:
  • Advantage: Private keys never exposed to connected (potentially compromised) host.
  • Physical Security: PIN protection, tamper-evidence features.
  • Considerations: Initial cost, physical security needs.
Example Modules:
  • Trezor (T/Safe 3)
  • Coldcard (Mk4/Q1)
  • Ledger (Nano/Stax)
  • ... (See Section VII for Matrix)
Paper Storage Unit

Physical key printout. Obsolete / High Risk Profile. Complex spending, fragile medium.

Inherent Vulnerabilities:
  • Spending Complexity: Partial spends difficult/unsafe (key exposure).
  • Physical Fragility: Susceptible to fire, water, decay, accidental damage.
  • Generation Risk: Requires secure, air-gapped generation environment.
  • Discovery Risk: Easily lost, stolen, or viewed accidentally.

Modern hardware wallets offer superior security and usability.

Air-Gapped Systems

Network-isolated signer + Online watch-interface. Data relay via SD/QR (PSBT). Extreme Network Isolation.

Operational Protocol:
  1. Generate unsigned TX (PSBT) on online interface.
  2. Transfer PSBT (SD/QR) to offline signing module.
  3. Authorize/Sign PSBT on secure offline module.
  4. Transfer signed PSBT back to online interface.
  5. Broadcast TX to network.
Tradeoffs:
  • Advantage: Maximum immunity to online malware/network attacks.
  • Disadvantage: Increased operational complexity, potential risks via transfer medium (SD card malware - less common for PSBTs).

Examples: Coldcard, Seedsigner.

// IV. Core Operations

Initialization Sequence

Secure seed generation *on device*. Backup seed (offline, steel). VERIFY BACKUP. Set strong credentials.

Initialization Protocol:
  1. Firmware Verification/Installation (HW wallets).
  2. Secure Seed Phrase Generation (Device RNG).
  3. Seed Phrase Backup: Accurate transcription, secure offline storage (e.g., SeedPlate, Cryptosteel).
  4. Backup Verification: Mandatory re-entry or check routine *before* receiving funds.
  5. Set Strong PIN / Password.
  6. Optional: Configure BIP-39 Passphrase (Advanced).
Receive Operation

Generate fresh address. VERIFY address on HW Trusted Display vs. host screen. Share Addr/QR.

Send Operation

Input recipient Addr (TRIPLE CHECK). Set Amount. Set Fee (sat/vB). Review. Confirm Addr/Amount on HW Screen.

Execution Flow:
  1. Recipient Address Input: Paste/Scan. Visually verify first/last chars vs. source.
  2. Amount Input.
  3. Fee Setting: Consult mempool.space. Balance urgency/cost.
  4. Software Review: Check all details on host screen.
  5. Hardware Confirmation: Verify address & amount match ON THE DEVICE'S SCREEN before authorizing signature.
  6. Broadcast Signed Transaction.
Balance/History Query

Use wallet interface. Optional: Public block explorer (mempool.space) - Input addresses only (privacy leak).

// V. Security Protocols :: OPSEC

Seed Phrase :: OPSEC Level MAX

[最重要 - CRITICAL] Your 12/24 words ARE your Bitcoin. Treat accordingly. Physical/Digital contamination = Total Loss.

Directive Alpha: ABSOLUTELY NO DIGITAL INPUT/STORAGE (No photos, typing, cloud, pwd mgrs). Assume ALL networked devices compromised. Directive Bravo: ZERO TRUST POLICY. NEVER SHARE SEED. "Support" requesting seed = IMMEDIATE SCAM DETECTED. Directive Charlie: BACKUP PHYSICALLY (STEEL RECOMMENDED), VERIFY ACCURACY, SECURE OFFLINE STORAGE (Redundant/Distributed Recommended).
Storage Strategy:
  • Recommended: Verified accurate transcription, Fire/Water/Corrosion proof steel plates, Multiple secure physical locations (consider geographic distribution), Obscurity.
  • Avoid: Plain paper (fragile), easily discoverable locations, single point of failure for backup.
  • Test Recovery Plan Periodically.
Hardware Module :: OPSEC

Source directly. Strong Credentials. VERIFY ALL TX DATA ON TRUSTED DISPLAY. Firmware Hygiene.

Standard Operating Procedures:
  • Acquisition Protocol: Source Direct from Manufacturer / Authorized Reseller ONLY. Verify tamper-evident seals.
  • Access Control: Implement strong, unique PIN. Understand and secure optional Passphrase if used (becomes part of seed).
  • Trusted Display Verification: MANDATORY check for every TX/Address generation. Confirms data against host malware (e.g., clipboard hijacks).
  • Firmware Integrity: Update only from official sources. Verify release notes. Have seed backup accessible *before* starting update. Consider delaying non-critical updates slightly.
  • Physical Security: Protect device from theft and unauthorized access ("Evil Maid Attack").
Host System & Network Hygiene

Use reputable wallet software (official sources). Keep OS/Security updated. Vigilant against Phishing. Avoid untrusted Networks.

Defense Layers:
  • Software Source: Download ONLY from official developer/manufacturer websites. Verify signatures if possible. Prefer open-source & well-audited.
  • System Updates: Keep Operating System and Anti-Malware definitions current.
  • Phishing Defense: Extreme skepticism towards unsolicited emails, DMs, popups requesting wallet interaction or credentials. Verify via known official channels.
  • Network Security: Avoid sensitive operations (seed recovery) on public/untrusted Wi-Fi. Consider VPN.
  • Clipboard Malware Awareness: Always re-verify pasted addresses visually. Use HW trusted display confirmation.
Backup, Recovery & Succession

Seed phrase IS the master key. Device is disposable. Have a tested recovery protocol. Plan for inheritance/incapacity.

Contingency Protocols:
  • Recovery Drill: Know the exact steps to restore wallet from seed on a compatible interface. Practice with testnet or small amounts *before* needed.
  • Estate / Succession Planning: CRITICAL to avoid permanent loss.
    • Securely documented instructions with trusted executor/lawyer.
    • Multisignature setups distributing control.
    • Specialized inheritance platforms (Vet carefully!). Examples: Casa, Unchained Capital.
  • Redundancy: Multiple secure backups mitigate single point of failure risk (theft, disaster).

// VI. Advanced Configurations

BIP-39 Passphrase

Optional "25th word" added to seed -> Creates separate hidden wallet instance. Security through obscurity/deniability.

Functionality:
  • Seed + Passphrase = Unique Wallet. Passphrase is never stored on device.
  • Use Case: Plausible deniability under duress (reveal non-passphrase wallet); extra security layer (requires seed + passphrase).
CRITICAL RISK:
  • Forgetting exact passphrase = PERMANENT FUND LOSS. NO RECOVERY. Must be backed up separately/memorized perfectly (risky). Case/space sensitive.
Multisignature (Multisig)

M-of-N key scheme (e.g., 2-of-3 signatures required). Distributes control & eliminates single points of failure.

Applications:
  • Enhanced Security: Requires compromise of multiple keys/devices/locations.
  • Shared Custody: Joint accounts, corporate treasury.
  • Robust Backup/Inheritance: Distribute keys among user, vault, lawyer, family.

Setup requires coordinator software (Sparrow, Specter, Nunchuk) using PSBTs.

  • Resource: Sparrow Multisig Guide
Connect to Full Node

Interface wallet with own Bitcoin Core node. Maximum Privacy & Trust Minimization.

Rationale:
  • Privacy: Avoids leaking address/transaction data to third-party servers.
  • Verification: Independently validate all network rules and transactions. No need to trust external providers.
  • Network Health: Contributes to decentralization.

Requires running node software (e.g., Bitcoin Core, Umbrel, Start9) and compatible wallet (Sparrow, Specter, Electrum + Server).

  • Resource: Node Implementation Options
Coin Control :: UTXO Selection

Manually select input UTXOs for spending. Advanced privacy & fee management technique.

Benefits:
  • Privacy: Avoids linking potentially unrelated UTXOs; breaks address reuse heuristics. Can segregate UTXO sources (e.g., KYC vs non-KYC - *note chain analysis limitations*).
  • Fee Optimization: Select UTXOs to minimize transaction size/cost.
  • Labeling: Track origin/purpose of specific UTXOs.

Requires wallet support (e.g., Electrum, Sparrow, Specter).

PSBTs :: Transaction Fragments

Partially Signed Bitcoin TX format (BIP-174). Standard for offline/multisig workflows.

Workflow Examples:
  • Air-Gap: Create unsigned PSBT online -> Transfer (SD/QR) -> Sign offline -> Transfer signed PSBT back -> Broadcast.
  • Multisig: Create PSBT -> Pass to signer 1 -> Sign -> Pass to signer 2 -> Sign -> ... -> Broadcast fully signed TX.
Fee Bumping :: RBF & CPFP

RBF (Replace-By-Fee): Replace own stuck TX w/ higher fee (BIP-125). CPFP: Spend stuck *incoming* TX w/ high fee.

Shamir Backup :: Seed Sharding

Split seed into M-of-N shares (SLIP-39). Requires threshold for recovery. Enhanced redundancy/security.

Tradeoffs:
  • Pros: Tolerate loss of some shares; attacker needs multiple shares.
  • Cons: Complex setup/recovery; less wallet support than BIP-39 (Trezor T primarily).
Testnet :: Simulation Grid

Parallel Bitcoin network using valueless coins. Safely test features, practice recovery, learn protocols.

// VII. Hardware Module Matrix

Comparison Vectors

Evaluate modules based on: Security Arch (SE/MCU), Source Code Auditability, Bitcoin Focus, Air-Gap Protocol, UX/UI, Advanced Feature Set, Cost, Vendor Trust.

Vector Analysis:
  • Security Arch: Secure Element (Dedicated, often closed) vs. Generic MCU (Open potential, relies on SW hardening).
  • Source Code: Open (Verifiable) vs. Closed (Trust-based). Firmware & Hardware.
  • Coin Support: Bitcoin-Only (Reduced attack surface) vs. Multi-Coin (Convenience, increased complexity).
  • Air-Gap: True air-gap (SD/QR/NFC) vs. USB/Bluetooth dependent.
  • UX/UI: Interface simplicity, screen quality, input method.
  • Features: Passphrase, Multisig (PSBT), Coin Control, Shamir etc. support level.
  • Cost.
  • Vendor Reputation: History, security posture, community trust.

// VIII. Threat Vectors & Failure Modes

Seed Phrase Compromise/Loss

FAILURE MODE #1: Physical loss/damage, incorrect backup transcription, backing up wrong initial seed, lost/forgotten passphrase.

Phishing / Social Engineering

FAILURE MODE #2: User Deception -> Seed revealed to fake site/app/support. Results in immediate, total loss.

Device/Software Failure

Forgotten PIN (requires seed), accidental wipe without verified backup, rare critical software bugs.

Host System Malware

Clipboard hijacker (verify HW screen!), keyloggers (risk SW), fake wallet apps.

Physical Compromise

Seed backup theft/discovery, unsecured HW theft (PIN/Passphrase=defense), coercion, insider theft.

Supply Chain / Bad Vendor

Tampered device (buy direct!), using flawed/malicious/unknown wallet interfaces.

Succession Failure

Lack of estate planning -> Funds permanently inaccessible after owner death/incapacity.

// IX. Terminology Lexicon

Protocol Glossary

Definitions for core Bitcoin & Wallet terminology.

Address
Identifier used to receive Bitcoin (e.g., starts with 1, 3, bc1q, bc1p).
Air Gap
Keeping a device (signer) physically isolated from network connections.
BIP-32
Standard for Hierarchical Deterministic (HD) wallets, deriving keys from a master seed.
BIP-39
Standard for mnemonic seed phrases (12/24 words) representing the master seed.
BIP-44/84/86
Standards defining derivation paths for different address types (Legacy, SegWit, Taproot).
BIP-125
Replace-By-Fee (RBF) standard.
BIP-174
Partially Signed Bitcoin Transaction (PSBT) format standard.
Bitcoin Core
The primary, reference implementation of the Bitcoin full node software.
Block Explorer
Website to view blockchain data (transactions, addresses, blocks).
Blockchain
The distributed, public ledger recording all Bitcoin transactions.
Cold Storage
Storing private keys offline.
Coin Control
Manually selecting specific UTXOs for spending.
Confirmation
Inclusion of a transaction in a block on the blockchain.
CPFP
Child-Pays-For-Parent. Spending an unconfirmed input with a high fee to accelerate parent confirmation.
Custodial
A third party holds your private keys (e.g., an exchange).
Derivation Path
Instructions for deriving specific keys/addresses from the master seed (e.g., m/84'/0'/0'/0/0).
Electrum
Popular, feature-rich software wallet.
Fee Rate (sat/vB)
Transaction fee measured in satoshis per virtual byte.
Full Node
Software that independently validates all Bitcoin transactions and blocks.
Hardware Wallet
Physical device securing private keys offline.
HD Wallet
Hierarchical Deterministic wallet, derives keys from a single seed (BIP-32).
Hot Wallet
Wallet where keys are stored on an internet-connected device.
Keys
Private Key (secret, for signing) and Public Key (sharable, for receiving).
Ledger
Popular hardware wallet manufacturer.
Lightning Network
Layer-2 protocol for fast, cheap Bitcoin transactions (uses different wallet types/concepts).
Mempool
Pool of unconfirmed transactions waiting for miners.
Multisignature (Multisig)
Requires multiple keys (M-of-N) to authorize a transaction.
Non-Custodial
You control your own private keys/seed phrase.
Passphrase (BIP-39)
Optional secret word/phrase added to seed, creating a hidden wallet ("25th word").
Private Key
The secret data allowing spending of Bitcoin.
PSBT
Partially Signed Bitcoin Transaction format.
Public Key
Cryptographic key derived from private key, used to generate addresses.
QR Code
Square barcode often used to represent addresses or transaction data.
RBF
Replace-By-Fee. Bumping the fee of an unconfirmed transaction.
Satoshis (Sats)
Smallest unit of Bitcoin (1 BTC = 100,000,000 sats).
Secure Element (SE)
Tamper-resistant chip designed to securely store cryptographic keys.
Seed Phrase
12/24 word mnemonic backup for the master private key (BIP-39).
Shamir Backup (SLIP-39)
Splits seed into multiple shares, requiring a threshold for recovery.
Signature
Cryptographic proof generated using a private key to authorize a transaction.
Software Wallet
Wallet application running on a computer or phone.
Sparrow Wallet
Desktop software wallet focused on security and privacy.
Specter Desktop
Desktop software wallet focused on multisig and hardware wallet interaction.
Taproot
Recent Bitcoin protocol upgrade enhancing privacy and scripting capabilities (P2TR addresses).
Testnet
Parallel Bitcoin network for testing with valueless coins.
Trezor
Pioneering hardware wallet manufacturer.
UTXO
Unspent Transaction Output. An individual "chunk" of bitcoin.
Watch-Only Wallet
Wallet loaded with public keys/addresses to monitor balance/transactions, but cannot spend.
