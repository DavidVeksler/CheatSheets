Modern Source Control Cheatsheet

An interactive guide using a history tree metaphor.

Foundational Concepts

DVCS Philosophy

Distributed Version Control Systems (DVCS) like Git & Mercurial give every developer a full repository copy, enabling offline work, faster operations, and flexible workflows.

Core Idea

Unlike older Centralized VCS (like SVN or CVS) where developers checked out files from a single central server, DVCS provides each user with a complete, independent copy (clone) of the entire repository history.

Key Advantages
  • Offline Work: Commit, branch, view history, and merge without needing a network connection.
  • Performance: Most operations (commit, branch, merge, diff) are local and thus significantly faster.
  • Redundancy: Every clone is effectively a full backup of the repository.
  • Workflow Flexibility: Enables complex branching strategies and collaboration models (e.g., pull requests).
  • Collaboration: Easily share changes between any two repositories, not just client-server.
Shift from Centralized

The move to DVCS addressed key limitations of CVCS, particularly around branching/merging complexity, performance bottlenecks, and reliance on network connectivity.

Repository (Repo)

A collection of files and the history of their changes. In DVCS, you have a local copy and interact with remote copies (e.g., on GitHub).

What it Contains

A repository stores all the files belonging to a project, plus the complete history of modifications made to those files over time. This history is typically stored as a series of commits.

Local vs. Remote
  • Local Repository: The full copy residing on your own computer. This is where you do your work: edit files, commit changes, create branches.
  • Remote Repository: A copy hosted on a server (e.g., GitHub, GitLab, a company server). Used for collaboration, backup, and sharing code with others. Common remote names include origin.
Operations

You typically clone a remote repository to create your local copy, pull changes from a remote, and push your local changes to a remote.

Commit

A snapshot of your project's files at a specific point in time, saved to the repository's history. Each commit has a unique ID and a message.

Purpose

Commits are the fundamental building blocks of a project's history. They represent saved states that you can reference, revert to, or compare against.

Components
  • Snapshot: Records the state of all tracked files at that moment. DVCS like Git often store this efficiently, only recording changes relative to previous commits.
  • Metadata: Includes the author, committer, timestamp, and a unique identifier (e.g., SHA-1 hash in Git).
  • Parent(s): Points to the preceding commit(s). A standard commit has one parent; a merge commit has two or more.
  • Commit Message: A description written by the author explaining the changes made in the commit. Crucial for understanding history.
Workflow (Git Example)

Typically, you modify files, stage the specific changes you want to include (git add), and then commit them with a message (git commit).

Branch

An independent line of development. Branches allow you to work on features or fixes without affecting the main codebase (e.g., main or master).

Concept

Think of a branch as a movable pointer to a specific commit. When you create a branch, you create a new pointer. As you make commits on that branch, the pointer moves forward.

Why Use Branches?
  • Isolation: Develop features, experiment, or fix bugs without destabilizing the main line of code.
  • Parallel Development: Multiple team members can work on different features simultaneously on separate branches.
  • Organization: Keep related changes grouped together (e.g., a 'feature/user-login' branch).
  • Workflow Enablement: Basis for workflows like Gitflow or GitHub Flow, often involving Pull Requests.
Common Operations

Creating a branch (git branch feature-x), switching to it (git checkout feature-x or git switch feature-x), making commits, and eventually merging it back.

Merge

The process of combining changes from different branches back into one. Resolves differences between the development histories.

Goal

To integrate work done on a separate branch (e.g., a feature branch) into another branch (e.g., main).

How it Works

VCS tools analyze the histories of the branches being merged.

  • Fast-Forward Merge: If the target branch hasn't diverged (no new commits), the target branch pointer simply moves forward to the source branch's latest commit. Simple and clean history.
  • Three-Way Merge: If both branches have diverged, the VCS finds a common ancestor commit and combines the changes from both branches since that ancestor. This typically creates a new merge commit with two parents.
Merge Conflicts

Occur when both branches modified the same part of the same file differently. The VCS cannot automatically decide which change to keep, requiring manual intervention by the developer to resolve the conflict before completing the merge.

Alternatives

Rebasing (e.g., git rebase) is another way to integrate changes, rewriting history to make it appear linear. Has different tradeoffs regarding history clarity and collaboration.

Distributed VCS

Git

The dominant DVCS, known for speed, flexibility, powerful branching/merging, and a vast ecosystem. Features a staging area.

Philosophy & Core

Designed by Linus Torvalds for Linux kernel development. Focuses on performance, data integrity (content-addressable storage using SHA-1), and support for non-linear workflows (branching).

Key Features
  • Staging Area (Index): Intermediate step between working directory and repository history. Allows crafting precise commits.
  • Branching Model: Extremely lightweight and fast branching and merging.
  • Distributed Nature: Full repository locally, facilitating offline work and peer-to-peer sharing.
  • Performance: Most operations are very fast as they are local.
  • Flexibility: Supports diverse workflows (Gitflow, GitHub Flow, etc.). Commands like rebase allow history manipulation (use with care).
Common Commands

git clone, git add, git commit, git status, git log, git branch, git checkout/git switch, git merge, git pull, git push, git fetch, git rebase.

Strengths
  • Industry standard, huge community, extensive tooling and platform support (GitHub, GitLab, etc.).
  • Excellent performance for most operations.
  • Powerful and flexible branching/merging capabilities.
  • Staging area allows granular control over commits.
Tradeoffs
  • Can have a steeper learning curve compared to Mercurial initially (especially concepts like the staging area, reset, rebase).
  • Command-line interface can feel complex with many options.
  • Handling very large binary files can require extensions (Git LFS).
Use Cases

Software development of all sizes, open-source projects, infrastructure as code, documentation, anywhere collaborative text-based file tracking is needed.

Mercurial (Hg)

A DVCS focused on simplicity, ease of use, and performance. Often considered more intuitive than Git initially.

Philosophy & Core

Designed with user-friendliness and consistency in mind. Aims to provide powerful DVCS features with a simpler command set and conceptual model compared to Git.

Key Features
  • Simpler Interface: Commands are often seen as more consistent and intuitive. No direct equivalent of Git's staging area by default (commits typically include all tracked changes).
  • Branching Concepts: Supports multiple ways to handle branches (named branches, bookmarks, anonymous heads), which can be confusing but offers flexibility. Named branches are permanent markers in history.
  • Extensibility: Features can be added via extensions (e.g., rebase, largefiles support similar to Git LFS).
  • Performance: Generally excellent performance, competitive with Git.
  • Windows Support: Historically known for strong native Windows support.
Common Commands

hg clone, hg add, hg commit (or hg ci), hg status, hg log, hg branch, hg update (switches branches/commits), hg merge, hg pull, hg push, hg heads.

Strengths
  • Easier initial learning curve for many users.
  • Consistent command-line interface.
  • Good performance.
  • Built-in web interface (hg serve).
Tradeoffs
  • Smaller community and ecosystem compared to Git.
  • Less prevalent on major hosting platforms (though Bitbucket has excellent support).
  • Branching model (especially named branches vs bookmarks) can be confusing.
  • Some advanced operations require enabling extensions.
Use Cases

Software development (especially where ease of use is prioritized), large projects (used by Facebook/Meta internally), teams comfortable with its branching models.

Hosting Platforms

GitHub

The most popular platform for hosting Git repositories. Strong focus on collaboration, open source, and developer community features.

Core Offerings

Git repository hosting, Pull Requests (code review/merge workflow), issue tracking, project management (Projects), CI/CD (GitHub Actions), package hosting (Packages), wikis, security scanning (Dependabot, Code Scanning), large community.

Key Features
  • Pull Requests: Core collaborative code review and merging mechanism.
  • GitHub Actions: Powerful, integrated CI/CD and workflow automation platform.
  • Community Focus: Hub for open-source projects, social coding features (following, starring).
  • Marketplace: Integrations with numerous third-party developer tools.
  • Codespaces: Cloud-based development environments.
  • Copilot: AI pair programmer integration.
Strengths
  • Massive user base and community effect.
  • Excellent UI/UX, generally intuitive.
  • Powerful and flexible GitHub Actions.
  • Generous free tier for public and private repositories.
  • Strong focus on security features.
Tradeoffs
  • Owned by Microsoft (a concern for some).
  • CI/CD minutes/storage limits on free/lower tiers can be restrictive for large projects.
  • Project management features less mature than specialized tools like Jira.
GitLab

An integrated DevOps platform built around Git hosting. Offers a complete toolchain from planning to monitoring. Self-hosting option available.

Core Offerings

Git repository hosting, Merge Requests, issue tracking with boards, powerful integrated CI/CD (GitLab CI/CD), container registry, security scanning (SAST, DAST, etc.), monitoring, planning tools (Epics, Roadmaps), wiki.

Key Features
  • Single Application: Aims to provide the entire DevOps lifecycle in one platform.
  • GitLab CI/CD: Highly integrated, powerful, and configurable CI/CD using a .gitlab-ci.yml file.
  • Self-Managed Option: Can be installed on your own infrastructure (Community Edition is open source).
  • Security Focus: Comprehensive suite of security scanning tools integrated into the development workflow (DevSecOps).
  • Auto DevOps: Opinionated pipeline for automating build, test, deploy, and monitoring.
Strengths
  • All-in-one DevOps platform reduces toolchain complexity.
  • Mature and powerful built-in CI/CD.
  • Strong security features integrated throughout.
  • Viable self-hosting option with the open-source Community Edition.
  • Transparent development process.
Tradeoffs
  • Can feel complex due to the sheer number of features.
  • UI can sometimes be less polished or intuitive than GitHub.
  • Free tier CI/CD limits can be hit quickly.
  • Self-hosting requires operational overhead.
Bitbucket Cloud

Atlassian's Git (and historically Mercurial) hosting platform. Known for excellent integration with other Atlassian tools like Jira and Trello.

Core Offerings

Git repository hosting (Mercurial support deprecated for new repos), Pull Requests, integrated CI/CD (Bitbucket Pipelines), code search, issue tracking integration (Jira), project tracking integration (Trello).

Key Features
  • Atlassian Ecosystem Integration: Seamless workflow between Bitbucket, Jira (issue tracking), Trello (boards), Confluence (documentation), etc.
  • Bitbucket Pipelines: Integrated CI/CD configured within the repository (bitbucket-pipelines.yml).
  • Code Insights: Surface information from scanning/testing tools directly in Pull Requests.
  • Free Tier: Offers free private repositories for small teams (up to 5 users).
Strengths
  • Unbeatable integration if you're heavily invested in the Atlassian suite (Jira especially).
  • Competitive pricing, potentially cheaper for small teams needing private repos than GitHub historically.
  • Simple and effective built-in CI/CD with Pipelines.
  • Clean UI.
Tradeoffs
  • Smaller community compared to GitHub.
  • CI/CD (Pipelines) may be less flexible or powerful than GitHub Actions or GitLab CI for complex scenarios.
  • Less focus on open-source community features.
  • Mercurial support is effectively legacy.
Use Cases

Teams using Jira for project management, organizations standardized on Atlassian tools, small teams needing cost-effective private repositories.

Azure DevOps Repos

Microsoft's Git hosting service within the broader Azure DevOps suite. Strong integration with Azure cloud and other DevOps services.

Core Offerings

Part of Azure DevOps Services which includes: Azure Repos (Git hosting, TFVC also supported), Azure Pipelines (CI/CD), Azure Boards (Agile planning, work item tracking), Azure Test Plans, Azure Artifacts (package management).

Key Features
  • Integrated Suite: Provides tools across the development lifecycle, similar to GitLab but within the Microsoft Azure ecosystem.
  • Azure Pipelines: Very powerful and flexible CI/CD system, supports building/deploying anywhere (not just Azure). YAML or classic visual editor.
  • Azure Boards: Mature Agile planning and work tracking tools.
  • Enterprise Focus: Strong support for permissions, policies, auditing, and integration with Azure Active Directory.
  • TFVC Support: Also supports Microsoft's legacy centralized VCS (Team Foundation Version Control).
Strengths
  • Excellent integration with Azure cloud services.
  • Mature and powerful CI/CD (Azure Pipelines).
  • Comprehensive work tracking features (Azure Boards).
  • Strong enterprise governance and security features.
  • Generous free tier for small teams and open source projects.
Tradeoffs
  • UI can feel complex and sometimes dated compared to GitHub/GitLab.
  • Less prominent in the open-source community than GitHub.
  • Can feel heavily tied to the Microsoft ecosystem (though Pipelines can deploy anywhere).
  • Split between different services (Repos, Pipelines, Boards) can feel less unified than GitLab's approach.
Use Cases

Organizations heavily invested in Microsoft Azure, enterprise teams needing strong governance and planning tools, teams using TFVC, projects requiring complex CI/CD pipelines.

Key Workflows

Branching Strategies

Defined patterns for using branches to manage development, releases, and fixes. Examples: Gitflow, GitHub Flow, GitLab Flow.

Common Strategies
  • Gitflow: A more complex model with dedicated branches for features (feature/*), releases (release/*), hotfixes (hotfix/*), development (develop), and production (main/master). Good for projects with scheduled releases.
  • GitHub Flow: Simpler model. main is always deployable. New work is done on feature branches created from main. Changes are merged back into main via Pull Request and deployed immediately. Optimized for continuous delivery.
  • GitLab Flow: Similar to GitHub Flow but adds optional environment branches (e.g., production, staging) or release branches, providing more flexibility between continuous delivery and versioned releases.
  • Trunk-Based Development: Developers work in short-lived branches or commit directly to the main line (trunk/main), relying heavily on feature flags and automated testing. Requires mature CI/CD practices.

The best strategy depends on team size, release cadence, deployment practices, and tooling.

Pull/Merge Requests

A mechanism on hosting platforms to propose changes from a branch, facilitating code review, discussion, automated checks, and controlled merging.

Process Overview
  1. Developer creates a feature branch, makes commits, and pushes it to the platform.
  2. Developer opens a Pull/Merge Request (PR/MR) targeting the main branch (e.g., main).
  3. Team members review the code changes, leave comments, and request modifications.
  4. Automated checks (CI builds, tests, linters, security scans) run against the proposed changes.
  5. Once approved and checks pass, the changes are merged into the target branch (often directly via the platform's UI).
Benefits
  • Code Quality: Facilitates peer review to catch bugs, improve design, and ensure consistency.
  • Knowledge Sharing: Team members learn from each other's code and provide feedback.
  • Gatekeeping: Ensures changes meet standards and pass tests before entering the main codebase.
  • Discussion Record: Provides a history of decisions and discussions related to the changes.
CI/CD Integration

VCS triggers Continuous Integration (automated build/test) and Continuous Delivery/Deployment pipelines, automating the software delivery process.

Continuous Integration (CI)

The practice of frequently merging code changes into a central repository, after which automated builds and tests are run.

  • Trigger: Typically runs on every push to a branch or opening/updating a PR/MR.
  • Goal: Detect integration errors early by automatically building the software and running automated tests (unit, integration).
  • Benefit: Faster feedback loop, reduced integration problems, improved code quality.
Continuous Delivery (CD) / Continuous Deployment (CD)

Extends CI by automating the release process.

  • Continuous Delivery: Automatically builds, tests, and prepares code changes for release to production. The final deployment to production might still be a manual step.
  • Continuous Deployment: Automatically deploys every validated change to production without manual intervention.
  • Trigger: Typically runs after CI succeeds on specific branches (e.g., main) or merges.
  • Benefit: Faster, reliable releases; reduced manual deployment effort and errors.
Role of VCS

VCS events (push, merge, tag creation) act as the primary triggers for CI/CD pipelines provided by platforms like GitHub Actions, GitLab CI/CD, Bitbucket Pipelines, Azure Pipelines, or dedicated tools like Jenkins.

Choosing Tools

System & Platform Selection

Choose Git unless you have specific reasons for Mercurial. Select a Platform based on ecosystem integration (Atlassian, Microsoft), desired features (CI/CD, security), community needs, and self-hosting requirements.

Choosing the VCS System (Git vs. Mercurial)
  • Default Choice: Git is the overwhelming industry standard. Choose Git for maximum compatibility, community support, tooling, and platform options.
  • Consider Mercurial if: Your team strongly prefers its perceived simplicity, you need its specific branching model, or you are contributing to an existing Mercurial project (e.g., some parts of Meta). Be aware of declining platform support.
Choosing the Hosting Platform (GitHub vs. GitLab vs. Bitbucket vs. Azure DevOps)
  • Community & Open Source: GitHub is the leader. Its Actions CI/CD is very popular.
  • Integrated DevOps & Self-Hosting: GitLab excels if you want an all-in-one solution or need to host it yourself. Strong built-in CI/CD and security focus.
  • Atlassian Ecosystem: Bitbucket is the best choice if you rely heavily on Jira and other Atlassian tools.
  • Microsoft Ecosystem & Enterprise: Azure DevOps is ideal for teams deep in Azure, needing enterprise governance, or leveraging its mature Boards and Pipelines.
  • CI/CD Needs: Evaluate the specifics of GitHub Actions, GitLab CI/CD, Bitbucket Pipelines, and Azure Pipelines based on your complexity and integration requirements.
  • Pricing & Tiers: Compare free tier limitations (users, private repos, CI/CD minutes, storage) and paid plan costs based on your team size and usage.
  • User Interface & Experience: Team preference for the platform's look, feel, and ease of use can be a factor.
Polyglot Approach?

While possible to use different platforms for different projects, most organizations standardize on one primary hosting platform for consistency, billing, and user management, while potentially integrating specialized tools from other ecosystems if needed.
